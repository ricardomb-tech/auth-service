# Story 1.5: Renovación con rotación y detección de reuso

Status: done

## Story

As a aplicación cliente,
I want canjear el Refresh Token por un par nuevo,
so that el usuario mantenga su sesión sin reautenticarse. (FR-4)

## Acceptance Criteria

1. Con un Refresh Token vigente y sin usar, `POST /auth/refresh` invoca `RefreshTokenUseCase`, que marca el token canjeado como usado (`used_at`) de forma atómica y emite un nuevo par Access+Refresh de la **misma** Familia de Tokens vía `TokenIssuer` (AD-2, AD-4, AD-13; FR-4).
2. Con un Refresh Token ya usado previamente (reintento de un token ya rotado — señal clásica de robo/reuso), toda la Familia de Tokens correspondiente queda revocada (`revoked_at` en todas sus filas) y la petición recibe el mismo 401 genérico que cualquier otro rechazo; cualquier canje posterior de otro miembro de esa misma familia también falla; el evento se registra en logs estructurados con `accountId` y `traceId`, sin exponer el token (FR-4, AD-4, AD-8, NFR-1, SM-2).
3. Con un Refresh Token expirado, ya revocado, o simplemente no reconocido (hash sin fila correspondiente), la petición recibe el mismo 401 genérico y se requiere un login completo — sin distinguir en la respuesta cuál de los tres motivos aplicó (FR-4, AD-8, mismo principio anti-enumeración que NFR-2 aplica a login/registro). **Nota de trazabilidad:** el AC original de epics.md para esta historia solo cubre explícitamente "expirado o revocado"; "no reconocido" (hash sin fila) se añade aquí por ser el mismo caso de fallo genérico y no requerir tratamiento propio — mismo principio, no una desviación de intención.

## Dev Notes

- **Continuidad con la Story 1.4 (done):** `RefreshToken.issue(accountId, familyId, ttl, clock)`, `RefreshToken.hashRawToken(rawToken)`, `TokenIssuer.issue(Account)`, `RefreshTokenRepository.save(...)`, la tabla `refresh_tokens` (V3, con `idx_refresh_tokens_family_id` e `idx_refresh_tokens_account_id`) y `AuthenticationFailedException`/`GlobalExceptionHandler` ya existen — reutilízalos, no los reimplementes. Los Dev Notes de la Story 1.4 dejaron explícitamente el terreno preparado para esta historia: *"el `familyId` se recibe como parámetro... porque... la futura rotación de la Story 1.5 reutilizará este mismo método con la familia existente"*; *"los métodos de búsqueda por hash y revocación de familia (necesarios para la Story 1.5) se añaden en esa historia, no aquí"*; *"no implementar todavía un `issue(Account account, UUID existingFamilyId)`... es exactamente el trabajo de `RefreshTokenUseCase` en la Story 1.5"*.
- **Por qué el `markUsedIfUnused` debe ser un `UPDATE` condicional atómico, no un `read → save()`:** ver Task 1 para la comparación detallada con `VerifyAccountUseCase`/`VerificationToken.consume()`, que sí usa el patrón más simple porque la concurrencia real sobre un click de verificación de email es prácticamente nula. Un endpoint de refresh es distinto: los SDKs cliente (apps móviles, SPAs) suelen reintentar automáticamente ante timeouts, así que dos peticiones concurrentes con el mismo Refresh Token crudo son un escenario realista, no un caso extremo teórico. Sin un `UPDATE ... WHERE used_at IS NULL` atómico, dos requests podrían leer el mismo token como "sin usar" y ambas tener éxito — lo cual rompe silenciosamente la garantía de AC #2 (RFC 9700, "OAuth 2.0 Security Best Current Practice", documenta esta clase de bug como el error de implementación más común en rotación de refresh tokens).
- **Revocación de familia como `UPDATE` masivo, no iteración:** `UPDATE refresh_tokens SET revoked_at = ? WHERE family_id = ? AND revoked_at IS NULL` en una sola sentencia — evita N round-trips a la base de datos y es la forma estándar recomendada para este patrón en PostgreSQL.
- **Cuenta no `ACTIVE` en el momento del refresh (Task 5, paso 6) — decisión tomada, no pedida explícitamente por ningún AC:** si una Cuenta fue `DISABLED`/`LOCKED` *después* de haber emitido un Refresh Token válido, esta historia rechaza el refresh (mismo 401 genérico) en vez de emitir un nuevo par para una Cuenta que ya no debería tener sesión activa. Es la lectura más coherente con AD-11 (deny-all por defecto) aplicada a la máquina de estados de Cuenta, y cierra un hueco obvio: sin este chequeo, una Cuenta desactivada por un Administrador seguiría pudiendo renovar su sesión indefinidamente mientras conserve un Refresh Token vigente, lo cual contradice el propósito de FR-11 (aunque la revocación proactiva de familias al desactivar una Cuenta es, en sí, trabajo de una historia de Epic 4 todavía no implementada). **Marcado como pregunta abierta para confirmar con el usuario** — ver mensaje de cierre de esta creación de historia.
- **Sin período de gracia para reintentos de red** — ver Task 5. Documentado aquí como referencia: si en el futuro se decide añadir uno, la investigación (Okta: 30s configurable 0-60s; Connect2id: 5s) sugiere una ventana de 5-30s donde el reuso del token *inmediatamente anterior* devuelve el mismo par ya emitido en vez de disparar la revocación — pero eso es trabajo de una historia futura, no de esta.
- **Relación con Story 1.6 (Logout, backlog):** 1.6 reutilizará el mismo `revokeFamily(familyId, revokedAt)` que esta historia introduce — no se necesita una revocación "por Cuenta completa" en 1.5 (ninguna AC lo pide; una Cuenta puede tener varias familias activas, una por dispositivo/sesión, y esta historia solo opera sobre la familia del token canjeado).
- **Riesgos heredados de la Story 1.4, no de esta historia (no resolver aquí):** el FK `refresh_tokens.account_id` sin política `ON DELETE` explícita sigue diferido (relevante solo cuando exista borrado de cuenta); la duplicación de generación de token entre `RefreshToken`/`VerificationToken` sigue siendo una decisión deliberada sin abstracción compartida.
